# 世界杯竞彩串关投注指南

## 第0步：参数确认（必须先于一切分析）

从用户请求中提取以下3个可变参数。**任何一个参数未被用户明确指定，必须先向用户提问确认后，才能进入后续流程。不可自行使用默认值跳过。**

| 参数 | 说明 | 用户可能的表述 | 确认提问示例 |
|------|------|---------------|-------------|
| **日期** | 分析哪天的赛事 | "明天"、"6月18日"、"周末" | "请问您想分析哪天的赛事？" |
| **玩法** | 胜平负 / 让球胜平负 | "胜平负"、"让球" | "请问玩法选胜平负还是让球胜平负？" |
| **串关数** | 几串1 | "2串1"、"3串1" | "请问做几串1？（当日可投赛事数为N场）" |

**不可变约束**：仅限世界杯赛事，不接受其他联赛。此条无需确认。

确认方式：一次性列出所有缺失参数，用简洁的提问向用户确认。例如：

- 用户只说"帮我看看世界杯" → 三项均缺失，一次性问：
  > 请确认以下信息：1️⃣ 哪天的赛事？2️⃣ 玩法选胜平负还是让球胜平负？3️⃣ 几串1？

- 用户说"明天世界杯2串1" → 日期和串关数已明确，只问玩法：
  > 玩法选胜平负还是让球胜平负？

- 用户说"明天世界杯让球胜平负" → 日期和玩法已明确，只问串关数：
  > 做几串1？（明天有N场世界杯赛事可投）

**参数解析规则**：
- "帮我看看明天的世界杯2串1" → 日期=明天, 玩法=❓需确认, 串关=2串1
- "推荐今天世界杯胜平负串关" → 日期=今天, 玩法=胜平负, 串关=❓需确认
- "6月18号世界杯让球3串1" → 三项全部明确，直接进入分析

## 核心流程

参数全部确认后，按以下5步严格执行，不可跳过任何步骤：

### 第1步：获取赛事数据

运行 `scripts/fetch_matches.py` 抓取赛事数据：

```bash
# 抓取所有可售赛事（默认不含已截止）
python3 scripts/fetch_matches.py

# 含已截止赛事
python3 scripts/fetch_matches.py --all

# 指定日期筛选
python3 scripts/fetch_matches.py --date 2026-06-18

# 指定日期 + 含已截止
python3 scripts/fetch_matches.py --date 2026-06-18 --all
```

脚本从 500.com 竞彩页面抓取，输出JSON格式，包含：
- 赛事编号、联赛、主客队
- 比赛时间、截止投注时间
- 让球数（正=主队受让，负=主队让球）
- 胜平负SP赔率（nspf，普通胜平负）
- 让球胜平负SP赔率（spf，让球后胜平负）
- 500.com分析页链接

**关于赔率字段**：
- `胜平负SP`（nspf）= 普通胜平负赔率，实力悬殊的比赛可能"未开售"，此时该字段为空
- `让球胜平负SP`（spf）= 让球胜平负赔率，这是竞彩核心玩法，绝大多数比赛都有

如脚本执行失败，用curl直接访问：
```bash
curl -s 'https://trade.500.com/jczq/' -H 'User-Agent: Mozilla/5.0' | iconv -f gb2312 -t utf-8
```

**筛选条件**：
- 仅保留 `data-simpleleague="世界杯"` 的赛事（不可变）
- 按用户指定日期筛选（脚本 `--date` 参数）
- 剔除已截止赛事（除非用户明确要求含已截止）
- 若指定日期世界杯赛事不足用户要求的串关场次数，自动向后续日期扩展，直到凑足（最低2场），并告知用户扩展了哪些日期的赛事

**无赛事兜底**：若抓取结果为空（当日无世界杯赛事可投），明确告知用户，并提示最近有赛事的日期。

### 第2步：获取深度分析数据

**架构原则：脚本粗提取 + LLM精解析**

500.com 分析页的 HTML 结构复杂且不稳定，纯正则匹配容易漏数据（如"澳门心水推荐"在 meta 标签和正文中各出现一次，正则只命到 meta 就以为没数据）。因此：

1. **脚本（代码）**：负责 curl 抓取、HTML→纯文本转换、按板块粗切分
2. **LLM**：读取纯文本，负责语义理解（哪个是正文、数据怎么结构化）

**使用 `scripts/fetch_analysis.py` 抓取并输出纯文本：**

```bash
# 默认：输出数据页完整纯文本（推荐，LLM 直接解析）
python3 scripts/fetch_analysis.py --fixtureid 1359212

# 抓全部3个页面（数据页+亚盘+欧赔）
python3 scripts/fetch_analysis.py --fixtureid 1359212 --page all

# 单独抓亚盘/欧赔页
python3 scripts/fetch_analysis.py --fixtureid 1359212 --page yazhi
python3 scripts/fetch_analysis.py --fixtureid 1359212 --page ouzhi

# JSON格式输出
python3 scripts/fetch_analysis.py --fixtureid 1359212 --page all --json

# 控制输出长度
python3 scripts/fetch_analysis.py --fixtureid 1359212 --max-chars 8000
```

脚本输出的是**整页纯文本**（已去 HTML 标签、解码实体、去导航噪音），不做板块切分。
LLM 直接读取全文，语义提取所有数据（排名、战绩、澳门心水、盘口等），无需代码预先切分。

**为什么不切分板块**：关键词切分（如按"澳门心水推荐"定位）本质是硬编码，网站改措辞就会断。
LLM 能在全文中找到任何数据，不需要代码预先切割，少一层预处理 = 少一个故障点。

| --page值 | 页面 | 包含数据 |
|----------|------|---------|
| shuju（默认） | 数据分析页 | FIFA排名、近期战绩、交锋记录、澳门心水推荐 |
| yazhi | 亚盘分析页 | 各公司亚盘初盘/即时盘口、水位变化 |
| ouzhi | 欧赔分析页 | 各公司欧赔初赔/即时赔、凯利指数 |
| all | 以上全部 | 以上所有数据 |

**兜底方案**：若 `fetch_analysis.py` 执行失败，可用 curl 直接抓取原始 HTML：
```bash
curl -s 'https://odds.500.com/fenxi/shuju-{fixtureid}.shtml' -H 'User-Agent: Mozilla/5.0' | iconv -f gb2312 -t utf-8
```
此时 LLM 需自行从 HTML 中提取信息（不推荐，仅作降级方案）。

**补充数据源：懂球帝**（需通过浏览器或搜索访问）

懂球帝(`dongqiudi.com`)是SPA架构，API有反爬，不能直接curl。用于补充500.com覆盖不足的数据：

- **世界杯数据页**: `https://www.dongqiudi.com/data?cid=61` — 可查看积分榜、球员榜、赛程
- **球队详情页**: `https://www.dongqiudi.com/team/{team_id}` — 阵容、伤病、停赛
- 使用 `$browser` skill 打开上述URL
- 或使用 `$tavily-search` 搜索 `"{球队名} 世界杯 伤病/阵容 site:dongqiudi.com"` 获取摘要

**补充数据源：体育资讯搜索**

使用 `$tavily-search` 搜索以下关键词，交叉验证和补充数据：
- `"{球队名} 世界杯 伤病"` — 伤停信息
- `"{球队名} 世界杯 大名单"` — 阵容信息
- `"{主队} vs {客队} 交锋记录"` — 历史对战
- `"世界杯 {日期} 赛程 赛前分析"` — 专家观点

### 第2.5步：获取阵容数据

500.com的预计阵容经常为空，**阵容数据必须通过脚本主动获取**。

**主数据源：懂球帝 API**（直接调用，无需浏览器）

运行 `scripts/fetch_squad.py` 获取阵容数据：

```bash
# 列出所有世界杯参赛队及ID
python3 scripts/fetch_squad.py --list-teams

# 获取指定球队阵容（支持模糊匹配）
python3 scripts/fetch_squad.py --team 西班牙

# 按小组批量获取（推荐）
python3 scripts/fetch_squad.py --group H

# 按懂球帝球队ID获取
python3 scripts/fetch_squad.py --team-id 1869

# JSON格式输出（供程序解析）
python3 scripts/fetch_squad.py --group H --json
```

脚本数据来源（优先级从高到低）：
1. **懂球帝 API**（主）— `member_v2` 接口：完整26人名单，按门将/后卫/中场/前锋分组，含球员姓名、年龄、所属俱乐部
2. **懂球帝 API**（辅）— `sample` 接口：FIFA排名、总身价、球场容量
3. **懂球帝 SSR**（备）— 世界杯数据页 `__NUXT__` 返回块：球队ID与分组映射

脚本自动识别五大联赛球员（★标记），统计五大联赛球员人数。

**备选数据源：tvly 搜索**（仅当懂球帝 API 不可用时）

```bash
tvly search "2026世界杯 {球队} 大名单 阵容" --max-results 2
```

**阵容评级标准**（详见 `references/squad_search.md`）：

| 评级 | 五大联赛球员 | 总身价 |
|------|------------|--------|
| ⭐⭐⭐⭐⭐ | ≥15人 | ≥8亿欧 |
| ⭐⭐⭐⭐ | 10~14人 | 4~8亿欧 |
| ⭐⭐⭐ | 5~9人 | 1~4亿欧 |
| ⭐⭐ | 1~4人 | 0.3~1亿欧 |
| ⭐ | 0人 | <0.3亿欧 |

**注意**：
- 懂球帝 API 无需认证，但需要正确的 Referer/Header
- 如 API 返回错误，降级使用 tvly 搜索
- 如无伤停报道，标注"暂无伤停报道"，不可杜撰

### 第3步：POWER-6 六维分析（赛事实证层）

**分析方法论详见 `references/analysis_methodology.md`，此处为执行摘要。**

对每场赛事，按POWER-6六维模型进行系统分析。**POWER-6负责逐场比赛的证据拆解、风险识别和加权评分，是后续三专家串关决策的共同事实基础。**

| 维度 | 核心问题 | 数据来源 |
|------|---------|---------|
| **战力鸿沟** | 双方实力差距有多大？ | FIFA排名、排名差等级 |
| **状态引擎** | 双方当前处于什么竞技状态？ | 近6场走势、近10场胜率/赢盘率 |
| **盘口密码** | 机构通过赔率暗示了什么？ | 亚盘初盘/即时盘、欧赔变动、凯利指数 |
| **交锋心结** | 历史交锋是否存在心理压制？ | 交锋记录、血脉压制/苦主效应 |
| **阵容博弈** | 球员层面是否有克制关系？ | 预计阵容、核心球员状态、伤停 |
| **赛程暗线** | 赛程安排是否有隐性影响？ | 休息天数、旅行距离、比赛时间、轮换可能 |

**评分规则**（10分制/维度）：

每个维度独立打分0~10分，按玩法对应的权重加权求和：

- **让球胜平负**：盘口密码35%、战力鸿沟15%、状态引擎15%、阵容博弈15%、交锋心结10%、赛程暗线10%
- **胜平负**：盘口密码25%、战力鸿沟20%、状态引擎20%、交锋心结15%、阵容博弈10%、赛程暗线10%

**评分→推荐映射**：
- ≥80分：强力推荐该方向（放入稳妥方案）
- 65~79分：推荐该方向（放入价值方案）
- 50~64分：方向有分歧（考虑平局/冷门）
- <50分：冷门信号（放入冷门方案）

**数据可用性自检（强制）**：抓取分析数据后，先读 `fetch_analysis.py` 输出末尾的【数据可用性自检】。标 ✗ 的维度，`desc` 写"数据暂缺"、`score` 按权重重分配，**严禁把方法论里的示例数字（0.85、0.76 等）当真实数据填入**。

**分析叙述风格**：
- 每场开头用1句话定性：碾压局 / 压制局 / 优势局 / 胶着局 / 均势局
- 引用数据时带状态评级emoji（🔥炽热/⚡上升/🔄波动/📉下滑/🥶冰封）
- 盘口分析使用专业术语（升盘不升水、降盘不降水等）
- 风险点用⚠️紧跟在推荐方向后
- 绝对禁止说"一定"、"必胜"、"稳赢"

### 第4步：三专家串关决策层

基于第3步各场 POWER-6 分析结果，输出**三专家方案体系**。三专家体系不替代 POWER-6，而是站在同一组 POWER-6 证据上做不同风格的取舍：

| 方案 | 专家 | 定位 | 决策偏好 | SP区间 |
|------|------|------|----------|--------|
| 🎯 稳妥方案 | Dr. Karl Vogel · 量化派 | 数据保守派 | 凯利指数、隐含概率、多公司赔率一致性 | 8~25 |
| 🔥 价值方案 | Johnny "The Handicapper" Liu · 盘口派 | 亚盘价值派 | 升降盘、水位异动、诱盘/阻上、走盘空间 | 25~80 |
| 💎 冷门方案 | Mia Carter · 消息派 | 情报冷门派 | 伤停、轮换、战意、赛程暗线、更衣室变量 | 80+ |

**组合原则**：
1. 不重叠风险：4串1中避免全选同一方向
2. 赔率平衡：稳妥方案SP不低于8
3. 逻辑互补：每个方案有清晰的叙事逻辑
4. 置信度标注：整体置信度 = 最低那场的置信度

### 第5步：输出报告

**核心原则：JSON是唯一数据源，HTML和文本都是JSON的渲染结果。**

1. 将 POWER-6 逐场分析和三专家方案整理为 `report_data.json`
2. 严格遵循 `references/report_schema.md` 的 JSON Schema
3. 调用 `scripts/generate_report.py` 渲染 HTML / 文本报告
4. 输出格式细节和三专家方案展示规范见 `references/report_templates.md`

```bash
python3 scripts/generate_report.py --schema
python3 scripts/generate_report.py --data report_data.json -o worldcup_report
python3 scripts/generate_report.py --data report_data.json -f html -o worldcup_report
```

## 关键约束

1. **仅限世界杯**：只分析 `data-simpleleague="世界杯"` 的赛事，忽略其他联赛（不可变）
2. **先确认后分析**：日期、玩法、串关数三项参数必须由用户明确指定，缺失任何一项都必须先提问确认，不可自行假设默认值
3. **玩法可选**：用户可指定胜平负或让球胜平负
4. **串关数可选**：用户可指定几串1，最低2串1
5. **日期可选**：用户可指定日期；不足场次时自动向后续日期扩展并告知用户
6. **数据真实性**：所有赔率、战绩、阵容数据必须来自实际查询，禁止杜撰
7. **必须标注来源**：每个数据点标注出处
8. **中文输出**：全中文报告
9. **赔率空值处理**：实力悬殊的比赛可能只有让球胜平负开售（胜平负未开售），反之极少数情况胜平负开售但让球胜平负未开售，需根据用户选的玩法检查对应赔率是否可用
10. **数据缺失降级**：某维度数据暂缺时，该维度评分标注"数据暂缺"，权重按比例重分配给其余维度
11. **方法论一致**：分析必须遵循 `references/analysis_methodology.md` 中的POWER-6六维模型和评分体系，不得自行发明维度或权重
12. **格式严格**：输出必须按上述报告模板，不得省略任何section，不得自行增加未定义的section
13. **JSON优先**：分析结果必须先组织为符合Schema的JSON数据，再通过 `scripts/generate_report.py` 渲染为HTML或文本。禁止直接输出未经验证的文本报告，确保HTML与文本内容100%一致
