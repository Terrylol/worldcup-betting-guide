# 数据解析稳健性评估与改进路线

本项目当前采用“脚本粗提取 + LLM 精解析”的架构：脚本负责抓取网页、转码、去噪和输出原始文本/结构化基础字段；LLM 负责读取文本中的盘口、赔率、战绩、伤停等语义信号。

## 当前脆弱点

### 1. 500.com 赛事列表依赖 HTML 属性与固定顺序

`scripts/fetch_matches.py` 通过正则提取 `<tr class="bet-tb-tr">` 和 `data-*` 属性，优点是轻量、无需浏览器；风险是：

- 页面类名或属性名一改，解析会直接为空。
- `data-type/data-value/data-sp` 的属性顺序如果变动，当前赔率正则可能漏抓。
- 编码固定优先 `gb2312`，如果站点改为 UTF-8 或返回压缩/风控页，需要更明确的异常识别。

建议：

- 改用 `html.parser` 或 `BeautifulSoup` 解析标签属性，不依赖属性顺序。
- 对抓取结果做结构校验：世界杯赛事数、每场 fixtureid、让球数范围、SP 三项完整性。
- 当结构化解析失败时，自动输出 `--raw-text` 并提示 Agent 进入 LLM 兜底解析。

### 2. 500.com 分析页只做文本去噪，没有结构化锚点

`scripts/fetch_analysis.py` 的设计是正确的：不硬切板块，把语义交给 LLM。但风险是：

- `trim_noise()` 的噪音关键词可能误删有用标题或遗漏新导航内容。
- `--max-chars` 默认 8000，欧赔/亚盘页面长时可能截断关键公司均值、凯利或最低值。
- 页面返回“验证码/登录/异常访问”时，纯文本仍可能看似成功。

建议：

- JSON 输出增加 `source_url`、`http_status`、`text_length_before_trim`、`text_length_after_trim`、`truncated` 字段。
- 对关键短语做健康检查：数据页应含“近期战绩/国际足联排名”，欧赔页应含“平均值/凯利”，亚盘页应含“盘口/水位”。缺失时标记 `quality: degraded`。
- `--page all` 时分别保存完整文本文件，LLM 读取文件而不是依赖终端截断输出。

### 3. Wikipedia 大名单解析依赖页面模板稳定

`scripts/fetch_squad_wiki.py` 依赖 `2026 FIFA World Cup squads` 的 wikitext 模板。风险是：

- 维基页面模板名、section 标题或球员字段变化会导致解析失败。
- 2026 赛前名单可能更新频繁，抓到的是快照而非固定版本。
- 国家名匹配依赖英文写法，中文队名到英文队名映射仍需人工/LLM 判断。

建议：

- 输出 `revid` 或 `oldid`，每次报告记录所用维基版本。
- 增加中文队名 → 英文队名 alias 表。
- 未找到球队时，自动 `--list-teams` 给候选，而不是静默失败。

### 4. 缺少缓存与可复现数据快照

当前脚本每次实时抓取，报告难以复现同一时刻数据。

建议：

- 增加 `data/cache/YYYY-MM-DD/fixtureid-page.txt` 缓存目录。
- 报告 JSON 写入 `sources[]`：URL、抓取时间、脚本版本、是否截断、数据质量状态。
- 支持 `--cache` / `--no-cache` / `--refresh` 参数。

## 推荐优先级

1. **先做结构化校验与质量标记**：成本低，能显著减少“看似成功但数据缺失”的情况。
2. **再做 HTML 属性顺序无关解析**：用标准库 `html.parser` 即可，不必引入重依赖。
3. **最后做缓存快照**：提高报告可复现性，也方便 debug 和分享。

## 不建议

- 不建议把 500.com 分析页强行全结构化。页面复杂且容易改版，纯脚本会变成大量脆弱正则。
- 不建议把 LLM 兜底去掉。当前“脚本粗提取 + LLM 精解析”是合理架构，应该增强质量检测，而不是过度硬编码。
